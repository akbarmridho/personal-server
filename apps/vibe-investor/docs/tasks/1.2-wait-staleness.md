# 1.2 Wait Staleness Tracking in Symbol Memory

Status: PLANNED
Priority: Iteration 1
Effort: Small
Impact: High — kills infinite WAIT loops by making wait duration visible and actionable
Depends on: 1.1 (extends the `active_recommendation` schema introduced there)

## Problem

When a WAIT or NO_CHASE is issued for a READY symbol, the agent has no memory of how long the wait has been active. Each session treats the WAIT as fresh. The same recommendation persists for weeks without progress.

## Changes

### `.opencode-config/skills/portfolio-management/SKILL.md` — desk-check review section

Add rule: when a WAIT or NO_CHASE is issued for a READY symbol, stamp the date and desk-check count into the symbol's memory file.

For every READY symbol carrying a WAIT, check `wait_desk_check_count`. If the same WAIT has persisted for 3+ desk-checks without the trigger level being tested, resolve one of:

1. Upgrade to actionable with adjusted entry
2. Downgrade to WATCHING with explicit reason
3. Renew the WAIT with fresh evidence and a new trigger level

"Same WAIT, no new evidence" is not a valid renewal.

### Symbol plan frontmatter

Extend `active_recommendation` (from 1.1) with staleness and retest tracking fields:

```yaml
active_recommendation:
  # ...existing fields from 1.1 (action, issued, horizon_expires, upgrade_trigger, downgrade_trigger, expiry_action)...
  wait_desk_check_count: 0
  retest_level: 10900
  retest_status: not_tested  # not_tested | tested_held | tested_failed
  retest_checked: null
```

## Files to Modify

## Acceptance

- Every WAIT-carrying READY symbol has `wait_desk_check_count` incremented each desk-check
- At count >= 3, the agent is forced to resolve (upgrade, downgrade, or evidence-backed renewal)
- `retest_level` and `retest_status` fields exist for 1.3 to consume
