# 1.3 Retest-Observed Check in TA UPDATE Mode

Status: PLANNED
Priority: Iteration 1
Effort: Small
Impact: High — kills goldfish retest behavior by forcing the agent to check whether the retest already happened
Depends on: 1.2 (reads and updates `retest_status` and `retest_checked` fields from the `active_recommendation` schema)

## Problem

The TA skill says "wait for retest at level X" but never checks whether price visited level X between sessions. The agent repeats the same recommendation without acknowledging that the retest may have already occurred (and held, or failed).

## Changes

### `.opencode-config/skills/technical-analysis/SKILL.md` — UPDATE requirements

Add mandatory pre-check to every UPDATE run:

Before repeating a prior "wait for retest at level X" recommendation, the UPDATE must check whether price visited level X since the prior review. Use `daily[]` and `intraday_1m[]` to determine:

- Tested and held: price reached the zone and showed acceptance/defense. Acknowledge explicitly. Evaluate whether the retest qualifies as the trigger the prior WAIT was waiting for. If yes, upgrade.
- Tested and failed: price reached the zone and broke through. Acknowledge explicitly. Downgrade the setup or update invalidation.
- Not tested: price did not reach the zone. State explicitly. If 3rd+ consecutive session without test, evaluate whether the setup is stale or entry zone needs adjustment.

Do not repeat a prior WAIT without stating which of these three outcomes occurred.

### Symbol plan update

Update `retest_status` and `retest_checked` fields (defined in 1.2) each session:

```yaml
active_recommendation:
  retest_status: tested_held  # not_tested | tested_held | tested_failed
  retest_checked: 2026-04-03
```

## Files to Modify

- `.opencode-config/skills/technical-analysis/SKILL.md` (UPDATE mode section)

## Acceptance

- Every TA UPDATE that references a prior WAIT explicitly states whether the retest was tested_held, tested_failed, or not_tested
- The agent never repeats "wait for retest at X" without checking the OHLCV data for the intervening period
- `retest_status` and `retest_checked` are updated in the symbol plan after each check
